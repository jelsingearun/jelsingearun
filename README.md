I'm a final-year **B.Tech Information Technology** student (Malla Reddy University, Hyderabad) who builds **full-stack, AI-assisted systems** end-to-end — from a FastAPI service and a MongoDB schema up through a React UI, with computer vision and developer-tooling work mixed in along the way. - **Full-stack civic-tech systems** — citizen-facing React apps backed by async FastAPI services and role-based access control
- **Computer-vision pipelines** — OpenCV-based verification services with a documented path to swap in a trained YOLOv8 detector
- **Developer automation tooling** — CLI utilities that scan, sanitize, and publish code safely, and ones that turn a codebase into review-ready documentation
- **Data analytics case studies** — Python/Excel/Jupyter workflows that turn raw CSVs into segmented, profit-oriented or trend-oriented insight

## 🎯 Problems I Like Solving

A recurring thread across my repos: **taking a manual, error-prone process and putting a structured pipeline in front of it** — manual pothole inspection → AI-assisted citizen reporting; manually re-typing project code into a report → an automated `.docx` extractor; manually vetting a folder for secrets before pushing it public → an automated scanner with sanitization built in.

## 📍 Current Focus

### 🚧 RoadCare — AI-Based Road Damage & Pothole Detection System
**[→ Repository](https://github.com/jelsingearun/RoadCare---AI-Based-Road-Damage-Pothole-Detection-System)**

**Purpose**
Manual road-damage inspection by municipal authorities is slow and reactive. RoadCare turns it into a citizen-sourced, AI-assisted, geospatially-clustered pipeline — citizens report, the system verifies and groups reports, authorities triage and act.

**Architecture Overview**
```
Citizen (React UI) → FastAPI API → Image Verification Layer → MongoDB
                                          │
                                          ▼
                                 H3 Hexagonal Clustering
                                          │
                                          ▼
                                Authority Dashboard (triage / repair tracking)
```
- **Input layer:** browser geolocation + image upload via the React/Vite citizen portal
- **Processing layer:** FastAPI validates and authenticates the submission (JWT), then an OpenCV-based verification service scores the image (with a documented, ready-to-enable path to swap in a trained `YOLOv8x` detector for higher accuracy)
- **Persistence layer:** MongoDB via the async `Motor` driver, with verification history stored separately from the report record for auditability
- **Spatial layer:** verified reports are bucketed into hexagonal cells with **Uber H3** to form stable, square-grid-bias-free risk zones
- **Output layer:** an authority dashboard for confidence-score review, risk-zone inspection, and repair-status tracking

**Tech Stack**
React · Vite · Axios · React Router · Tailwind CSS &nbsp;|&nbsp; FastAPI · Uvicorn · Pydantic · Motor &nbsp;|&nbsp; OpenCV · NumPy · Pillow · Ultralytics YOLOv8 (documented integration) &nbsp;|&nbsp; MongoDB &nbsp;|&nbsp; Uber H3 &nbsp;|&nbsp; pytest (unit, integration, geospatial, DB, and frontend E2E suites)

**Key Features**
- Citizen auth (signup/login) and authenticated, geotagged pothole reporting
- AI-assisted image verification with a configurable confidence threshold
- H3-based risk-zone clustering for proactive maintenance planning
- Role-based access control separating citizen and authority workflows
- Full test coverage across backend, geospatial logic, and frontend flows

**Engineering Value**
It's the clearest evidence of full-stack systems thinking in the portfolio: async API design, JWT-secured RBAC, a real (if intentionally lightweight and explainable) CV integration, a non-trivial geospatial algorithm, *and* self-aware documentation of its own limitations — low-light/blur sensitivity, and a non-transactional risk-zone rebuild window — which is the kind of honesty that reads well in a system-design interview.

### 🤖 GitHub Repo Automator
**[→ Repository](https://github.com/jelsingearun/github-repo-automator)**

**Purpose**
Publishing many local or zipped projects to GitHub safely is tedious and risky — secrets get committed, repos go out with no README/LICENSE/.gitignore. This CLI automates the entire prepare → secure → publish pipeline.

**Architecture Overview**
```
Input (Folder / ZIP / Single File)
        │
        ▼
  Security Scan  →  Secret Sanitization  →  Template Injection
        │
        ▼
 Git Init → Initial Commit → GitHub Repo Creation (via gh CLI) → Push
```

**Tech Stack**
Python · Git · GitHub CLI (`gh`) · regex + Shannon-entropy secret detection · static dangerous-function analysis · Discord/Slack webhook notifications

**Key Features**
- Bulk-converts folders, ZIP archives, or single files into individually pushed GitHub repos
- Custom secret scanner flags API keys, OAuth/JWT tokens, AWS credentials, and private keys, then auto-sanitizes them before commit
- Flags risky functions (`eval`, `exec`, `os.system`, `subprocess.Popen`, `pickle.load`)
- Auto-generates missing README, MIT LICENSE, `.gitignore` (language-aware), issue and PR templates
- Dry-run mode, organization-repo support, and a local `repo_automation.log` audit trail
- Documented benchmarks (e.g. ~3 minutes for 50 repositories on an 8-core/16GB reference machine)

**Engineering Value**
This is the strongest signal of security-conscious engineering in the portfolio — not just *building* a scanner, but documenting its own blind spots (false positives on high-entropy non-secrets, encoded/base64 secrets, binary files) and explicitly recommending it be paired with Bandit, Trivy, or Snyk rather than treated as a complete solution.

### 📄 Project Code Extractor
**[→ Repository](https://github.com/jelsingearun/project-code-extractor-main)**

**Purpose**
Academic project defenses and reports need the full source code documented — manually copy-pasting files into Word is slow and error-prone.

**Architecture Overview**
A Python CLI prompts for a project path and an output path, recursively scans the project, filters by a whitelist of extensions while skipping `node_modules` and minified files, then writes a structured `.docx` via `python-docx`, formatting each filename as a heading.

**Tech Stack**
Python · `python-docx`

**Key Features**
- Supports `.py .js .java .cpp .c .html .css .ts .kt .swift .php`
- Skips dependency folders and minified bundles automatically
- Graceful encoding-error handling and a file-count summary on completion

**Engineering Value**
A small tool, but a telling one — the generated `Project_Code_Documentation.docx` shows up *inside* the RoadCare repository itself, meaning this wasn't a one-off exercise but a utility actually built to support a larger project's deliverables.
